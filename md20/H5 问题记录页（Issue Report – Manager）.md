H5 问题记录页（Issue Report – Manager）

Wireframe · Text Version

一、页面定位（先把性质定死）

页面类型：即时记录页

使用场景：

巡店中发现问题

收银 / 制作过程发现问题

事发当下

页面原则：

快

不能模糊

不能事后补写

二、整体结构（单屏优先，必要可滚动）
┌──────────────────────────────┐
│ 顶部：页面标题               │
├──────────────────────────────┤

├──────────────────────────────┤
│ 问题说明（必填）             │

├──────────────────────────────┤
│ 提交按钮（固定底部）         │
└──────────────────────────────┘

三、顶部区域（Header）
← 返回      ⚠ 记录问题


只保留：

返回

标题

❌ 不允许选择日期

❌ 不允许查看历史

这个页面永远只记录“现在”

四、问题类型选择（必选 · 强约束）
模块标题
📌 问题类型（必选）

UI 结构（单选）
( ) 异常
( ) 货损
( ) 错做

类型说明（小字）

异常：流程 / 行为 / 规范问题

货损：原料、成品损耗

错做：制作错误、下错单

规则

必须选 1 个

不允许自定义类型

不允许多选

五、问题拍照（⭐ 核心，必填）
模块标题
📷 问题现场照片（必填）

UI 结构
[ 📸 拍照 ]


拍照后：

[ 照片 1 ] [ 照片 2 ] [ + ]

强约束规则

✅ 只能调用相机

❌ 禁止相册选择

✅ 至少 1 张

建议上限：5 张

无照片 = 问题不存在

六、问题说明（必填，但短）
模块标题
✏️ 问题说明（必填）

UI 结构
[ 输入框 ]

占位提示（引导用语）

简要说明发生了什么，不需要长文

规则

必填

不要求规范格式

不限制长度（但鼓励简短）

七、关联信息（可选，弱约束）

用于后续追责 / 统计，但不影响提交

UI 结构
👤 相关员工（可选）
[ 选择员工 ]

📋 关联巡店（自动 / 可选）
[ 当前巡店 / 无 ]

规则

可不选

不影响提交

系统可自动填充（如从巡店进入）

八、提交按钮（固定底部）
[ ✅ 提交问题记录 ]

按钮状态规则
条件	状态
未选类型	禁用
未拍照	禁用
未填说明	禁用
正在提交	loading
条件满足	可点击
九、提交后的系统行为（必须一致）

提交成功后：

弹出提示：

✔ 问题已记录


自动跳转：

店长首页
或

返回巡店页（如从巡店进入）

系统记录：

创建时间

创建人（店长）

关联巡店（如有）

十、明确禁止的行为（一定要拦）

❌ 回填历史问题

❌ 修改已提交问题

❌ 上传相册旧图

❌ 删除问题记录（仅老板可处理）

十一、验收标准（真实场景）

在嘈杂的店内环境：

单手操作

1 分钟内完成

不需要思考“怎么填”

如果能 → 页面合格。

十二、一句话总结

这是一个“留证据”的页面，
不是“写报告”的页面。
